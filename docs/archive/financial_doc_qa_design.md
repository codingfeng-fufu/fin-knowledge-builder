# 面向金融文档的问答系统完整设计方案

## 1. 原始聊天记录整理

基于最开始的讨论，可以把核心共识整理为以下几点：

1. 系统面向金融文档场景，用户会围绕文档提出各种复杂问题，不只是简单检索。
2. 问题类型至少包括：
   - 数值计算
   - 政策/条款判断
   - 担保链与关系分析
   - 多文档交叉核对
3. 系统一开始可以有一些基础默认模板，但模板不是核心竞争力。
4. 真正的核心是：把用户问题背后的处理逻辑不断沉淀下来，形成规则库。
5. 后续新用户提出类似问题时，系统应像 RAG 一样，把历史上学到的处理方法检索出来，再按已有步骤解决问题。
6. 大模型不应该直接一把梭回答，而应该接收拆解后的清晰小任务，逐步完成抽取、判断、计算、解释。
7. 模板层可以兼容 Ollama/Qwen 使用的 Go Text Templates 风格，但这只是提示词渲染层，不是系统核心抽象。
8. 更适合的中间层是“业务逻辑知识图谱”或“结构化规则库”，因为它更直观、更易展示、更便于长期维护。
9. 这个机制和 OpenClaw/skills 的思路相近，但我们的目标是从案例中自动学习出规则，而不是完全靠人工手写。
10. 前期可以组织同学们做“客户问题仿真 + 正确答案 + 解题步骤整理”，快速形成第一批规则学习样本。

一句话总结：

**我们要做的不是普通金融 RAG，而是一个能从案例中学习解题方法、把方法沉淀为规则资产、再驱动大模型分步执行的金融文档问答系统。**

## 2. 产品定义

### 2.1 产品定位

这是一个面向金融文档的智能问答与分析系统。系统的目标不是单纯“回答问题”，而是：

1. 从文档中定位事实和证据
2. 从规则库中召回历史上学过的处理方法
3. 把复杂问题拆成多个小步骤
4. 由大模型与工具协同执行这些步骤
5. 输出答案、证据、过程和可解释链路

### 2.2 典型问题类型

系统优先支持三类问题：

1. 数值计算类
   - 例如：统计授信余额、汇总逾期金额、计算担保覆盖率
2. 政策/条款判断类
   - 例如：某项安排是否符合监管要求、某条款是否触发限制
3. 关系/链路分析类
   - 例如：担保链分析、关联方穿透、上下游依赖关系

### 2.3 产品核心价值

和普通文档问答相比，本系统的价值不在“找到相关段落”，而在：

1. 能处理有业务逻辑的问题，而不是只做语义匹配
2. 能复用历史案例里已经验证过的处理方法
3. 能把专家经验沉淀成长期规则资产
4. 能随着案例增加持续变强，而不是每次重新发挥

## 3. 核心设计思想

### 3.1 文档提供事实，规则提供方法，大模型负责执行

这是整个系统最关键的设计原则：

1. 文档和事实库回答“数据在哪里、证据是什么”
2. 规则库回答“这个问题应该怎么做”
3. 大模型回答“如何执行某一步、如何解释结果”
4. 工具模块负责精确计算、过滤、图分析和校验

### 3.2 系统核心不是 prompt，而是规则资产

单个 prompt 只能解决一次问题；规则资产能解决一类问题。

所以系统应该先把业务逻辑结构化，再把它编译成 prompt 或 workflow，而不是把所有经验直接写死在 prompt 里。

### 3.3 知识图谱是目标形态，结构化规则是可执行形态

推荐采用“双层表示”：

1. 知识图谱/规则关系层
   - 用于展示、关联、检索、解释
2. 规则 DSL/结构化规则层
   - 用于执行、编排、版本管理

也就是说：

- 图谱负责表达和检索
- DSL 负责执行和落地

## 4. 总体架构

建议把系统拆成七层：

### 4.1 文档接入层

负责接入 PDF、Word、Excel、扫描件、公告、合同、制度文件等。

### 4.2 文档解析层

负责：

- OCR
- 版面分析
- 表格抽取
- 段落切分
- 页码和坐标映射

### 4.3 事实抽取层

从解析后的文档中抽取结构化事实：

- 公司/主体
- 金额/币种
- 日期/期限
- 指标/口径
- 条款/限制条件
- 担保关系/股权关系/交易关系

### 4.4 规则资产层

存储系统核心资产：

- 问题类型
- 规则定义
- 处理步骤
- 校验逻辑
- 规则版本
- 规则与案例的映射关系

### 4.5 召回与编排层

根据用户问题：

1. 识别意图和约束条件
2. 召回相关案例和规则
3. 生成步骤计划
4. 决定每一步使用 LLM 还是工具

### 4.6 执行层

执行具体步骤，包括：

- 信息抽取
- 逻辑判断
- 数值计算
- 图遍历
- 结果校验
- 最终答案生成

### 4.7 学习反馈层

把以下内容回流到规则资产层：

- 新案例
- 人工修正
- 规则命中效果
- 执行失败原因
- 规则边界补充

## 5. 规则知识图谱设计

### 5.1 为什么要用知识图谱做中间层

相比直接存一堆 prompt，知识图谱更适合金融业务逻辑沉淀：

1. 更直观，业务人员可以看懂
2. 更适合展示问题、步骤、条件之间的关系
3. 更方便做组合式复用
4. 更容易做版本管理和规则演进
5. 更方便从规则生成不同模型的 prompt

### 5.2 规则图谱中的节点类型

建议至少包含以下节点：

1. `QuestionType`
   - 数值统计、条款判断、关系分析
2. `Intent`
   - 求和、筛选、比较、归因、穿透、去重
3. `DocumentType`
   - 授信报告、合同、公告、审计报告、制度文件
4. `Entity`
   - 公司、子公司、担保人、被担保人、项目、产品
5. `Metric`
   - 余额、额度、逾期金额、利率、期限
6. `Condition`
   - 时间范围、币种、主体范围、合并口径
7. `MethodStep`
   - 抽取、过滤、关联、去重、汇总、校验
8. `Tool`
   - OCR、表格抽取、计算器、图分析、SQL 查询
9. `PromptTemplate`
   - 某一步骤对应的提示词模板
10. `ValidationRule`
   - 金额单位统一、时间范围检查、口径一致性检查
11. `Case`
   - 来源案例和验证案例

### 5.3 规则图谱中的关系类型

建议包括：

- `QuestionType -> requires -> MethodStep`
- `Intent -> maps_to -> MethodStep`
- `MethodStep -> uses -> Tool`
- `MethodStep -> implemented_by -> PromptTemplate`
- `ValidationRule -> checks -> MethodStep`
- `Rule -> derived_from -> Case`
- `Case -> validates -> Rule`
- `Rule -> similar_to -> Rule`

### 5.4 规则的结构化表示

建议每条规则至少包含：

- 规则编号
- 规则名称
- 适用问题类型
- 触发条件
- 适用文档类型
- 所需输入
- 处理步骤
- 校验规则
- 输出格式
- 版本号
- 状态
- 来源案例
- 适用范围
- 不适用边界

示例：

```yaml
rule_id: guarantee_balance_aggregate_v1
rule_name: 统计担保余额
question_types:
  - 数值统计
trigger_signals:
  - 担保余额
  - 担保责任余额
  - 在保余额
applicable_documents:
  - 授信报告
  - 担保合同
required_inputs:
  - subject_scope
  - time_range
  - currency_scope
steps:
  - step_id: extract_guarantee_records
    action: 抽取担保人、被担保人、余额、起止日期
    tool: ie_extractor
  - step_id: normalize_amount_unit
    action: 统一金额单位
    tool: normalizer
  - step_id: filter_by_effective_date
    action: 过滤指定时间范围内有效的担保记录
    tool: rule_filter
  - step_id: deduplicate_relations
    action: 对重复担保关系去重
    tool: entity_resolver
  - step_id: aggregate_balance
    action: 汇总担保余额
    tool: calculator
validation_rules:
  - amount_unit_consistency
  - duplicate_relation_check
output_schema:
  - final_value
  - evidence_list
  - calculation_trace
version: v1
status: published
```

## 6. 规则工厂 MVP

### 6.1 定位

规则工厂不是最终问答产品，而是系统的“知识生产线”。

其职责是：

1. 接收人工案例
2. 生成候选规则
3. 审核并发布规则
4. 支持规则被新问题召回
5. 记录反馈并更新规则

### 6.2 MVP 核心闭环

**案例入库 -> 候选规则生成 -> 人工审核 -> 规则发布 -> 新问题命中 -> 执行反馈 -> 规则更新**

### 6.3 MVP 必须具备的模块

1. 案例库
2. 规则 Schema / DSL
3. 候选规则生成器
4. 规则审核发布台
5. 规则召回模块
6. 反馈回流模块

### 6.4 MVP 最小数据对象

#### `Case`

- `case_id`
- `title`
- `question_text`
- `question_type`
- `document_types`
- `gold_answer`
- `evidence_list`
- `solution_steps`
- `review_status`

#### `Rule`

- `rule_id`
- `rule_name`
- `applicable_question_types`
- `trigger_signals`
- `applicable_documents`
- `required_inputs`
- `steps`
- `validation_rules`
- `output_schema`
- `version`
- `status`

#### `RuleCaseLink`

- `rule_id`
- `case_id`
- `relation_type`

#### `ExecutionTrace`

- `trace_id`
- `input_question`
- `matched_rules`
- `execution_steps`
- `final_result`
- `manual_correction`
- `failure_reason`

### 6.5 MVP 成功标准

一期成功不以“回答所有问题”为标准，而以以下指标为标准：

1. 能沉淀出第一批正式规则
2. 每条规则能追溯到来源案例
3. 新问题能召回规则并形成步骤计划
4. 人工修正能反向更新规则版本
5. 第二次处理类似问题时，比第一次更快、更稳

## 7. 规则学习样本设计

### 7.1 定义

规则学习样本不是普通问答对，而是：

**可追溯、可复现、可抽象成规则的业务案例。**

每个样本都应该回答五个问题：

1. 用户问了什么
2. 正确答案是什么
3. 证据在哪里
4. 这个答案是怎么做出来的
5. 这套做法能否抽象为规则

### 7.2 样本必须包含的内容

- 问题原文
- 问题类型
- 文档类型
- 标准答案
- 证据片段
- 解题步骤
- 计算或判断逻辑
- 校验点
- 适用范围
- 不适用边界

### 7.3 推荐字段

#### 基础信息

- `case_id`
- `title`
- `source_type`
- `question_text`
- `question_paraphrases`

#### 问题标签

- `question_type`
- `intent`
- `document_types`
- `target_entities`
- `target_metrics`
- `constraints`

#### 标准答案

- `gold_answer`
- `answer_format`
- `evidence_list`

#### 解题过程

- `solution_steps`
- `formula_or_logic`
- `validation_checks`
- `common_errors`

#### 规则抽象信息

- `candidate_rule_name`
- `applicable_scope`
- `non_applicable_scope`
- `reviewer`
- `review_status`

### 7.4 最关键字段：解题步骤

能否形成规则，关键取决于 `solution_steps` 是否写清。

建议每一步固定描述：

- 本步目标
- 输入
- 输出
- 使用的字段或工具

示例：

```yaml
solution_steps:
  - step_id: 1
    action: 识别主体范围
    input: 用户问题
    output: A集团及其子公司名单
  - step_id: 2
    action: 抽取担保记录
    input: 文档正文和表格
    output: 担保人、被担保人、余额、日期
  - step_id: 3
    action: 过滤时间条件
    input: 担保记录 + 2024年末
    output: 有效担保记录
  - step_id: 4
    action: 去重并汇总余额
    input: 有效担保记录
    output: 最终担保余额
```

### 7.5 样本类型

建议至少包含四类：

1. 标准样本
   - 高频、典型、最容易抽象规则
2. 改写样本
   - 同一规则的不同问法，训练召回鲁棒性
3. 边界样本
   - 特殊口径、特殊约束，用于补边界
4. 反例样本
   - 表面相似但不应套用同一规则，用于防误召回

### 7.6 样本生产角色

建议至少三种角色：

1. 出题人
   - 模拟真实客户问法
2. 解题人
   - 给出标准答案、证据和步骤
3. 审核人
   - 确认答案、证据、步骤和抽象是否正确

如果团队允许，再加：

4. 规则抽象人
   - 专门把样本转换成候选规则

## 8. 在线问答执行流程

### 8.1 处理总流程

用户提问后，系统按以下顺序处理：

1. 问题理解
   - 识别问题类型、约束条件、主体范围、时间范围、输出格式
2. 文档召回
   - 召回相关段落、表格、附注和结构化事实
3. 规则召回
   - 召回相似规则和历史案例
4. 步骤编排
   - 生成本次问题的步骤计划
5. 步骤执行
   - 每一步由 LLM 或工具执行
6. 结果校验
   - 对金额、时间、口径、证据一致性做检查
7. 答案生成
   - 输出结论、证据、计算过程和解释
8. 反馈回流
   - 记录是否修正、是否形成新规则或更新旧规则

### 8.2 执行层原则

不要让大模型负责所有事情。

建议：

1. LLM 负责
   - 问题理解
   - 规则选择
   - 条款归纳
   - 中间解释
2. 工具负责
   - 数值计算
   - 表格聚合
   - 图路径搜索
   - 去重
   - 规则校验

### 8.3 步骤粒度要求

每一步都应该满足：

1. 输入明确
2. 输出明确
3. 可单独验证
4. 可替换实现

不要设计一个“分析担保情况并给结论”的大步骤，而要拆成：

1. 抽取担保记录
2. 统一金额单位
3. 过滤时间条件
4. 建立关系图
5. 去重汇总
6. 输出结论与证据

## 9. 模板与大模型执行层设计

### 9.1 Go Text Templates 的角色

像 Ollama/Qwen 中使用的 Go Text Templates，本质上是：

**把结构化上下文渲染成最终 prompt 的模板引擎。**

它可以兼容，但不应该成为系统的核心抽象。

### 9.2 推荐的层次关系

建议采用四层结构：

1. 规则/图谱层
   - 描述应该做什么
2. 执行计划层
   - 描述按什么顺序做
3. Prompt 模板层
   - 描述如何把某一步翻译成模型输入
4. 模板引擎层
   - 负责变量渲染

### 9.3 为什么不直接把经验写成 prompt

因为直接写 prompt 会带来几个问题：

1. 难展示给业务方
2. 难维护和版本化
3. 难复用和组合
4. 难把案例稳定转换为资产

所以更合理的做法是：

**规则先结构化，再编译成 prompt。**


### 9.4 落地执行层的定位

规则学习到了，Skill 也创建好了，并不意味着 LLM 就能稳定把任务做对。

真正还需要一层“运行时执行编排层”，负责把规则、知识图谱、Skill 和当前问题上下文组织起来，生成一组受控、可验证、可回溯的小任务，再交给 LLM 和工具执行。

这一层的本质作用是：

**让 LLM 不是自由发挥，而是沿着规则和 Skill 规定好的轨道完成任务。**

### 9.5 完整执行链路

完整链路建议明确为：

**案例 -> 规则 -> KG -> Skill -> 执行编排层 -> LLM/工具执行 -> 反馈回流**

其中：

1. 案例层提供经验来源
2. 规则层沉淀处理逻辑
3. KG 层组织逻辑关系和可解释结构
4. Skill 层封装高频、稳定、可参数化的能力单元
5. 执行编排层负责把这些能力转化为本次问题的执行动作
6. LLM 和工具负责完成具体步骤
7. 反馈层负责把新经验再写回规则资产

### 9.6 执行编排层要解决的问题

这一层至少要解决六件事：

1. 理解当前问题
   - 识别问题类型、目标、约束条件、输出要求
2. 召回相关规则和 Skill
   - 只选择当前问题相关的规则子集和 KG 子图
3. 生成任务计划
   - 把复杂问题拆成步骤 DAG 或 workflow
4. 编译步骤任务
   - 把每一步转换成 LLM 可理解、可执行的受控任务
5. 调度工具和 Skill
   - 决定哪些步骤由 LLM 执行，哪些步骤由工具完成
6. 校验与重试
   - 当证据不足、格式错误、结果冲突时，触发重试、补充检索或人工介入

### 9.7 真正指导 LLM 的不是普通 Prompt，而是 Step Contract

建议不要把模板简单理解为“回答提示词”，而应该理解为：

**规则、Skill 和上下文在当前步骤上的执行合同。**

也就是说，运行时真正交给 LLM 的，不是一段松散文字，而是一个结构化任务约束对象。模板只负责把这个对象渲染成模型能理解的输入格式。

### 9.8 Step Contract 建议结构

每一步都建议包含以下要素：

1. 当前目标
   - 这一步到底要完成什么
2. 当前输入
   - 已知事实、文档片段、上一步结果
3. 适用规则
   - 当前必须遵守哪些规则和边界条件
4. 可用动作
   - 当前允许调用哪些工具和 Skill
5. 输出格式
   - 必须返回什么结构
6. 校验要求
   - 如何判断这一步结果是否有效
7. 失败策略
   - 证据不足、结果冲突时如何处理

示例：

```yaml
step_id: aggregate_guarantee_balance
goal: 统计 2024 年末 A 集团及其子公司的对外担保余额
skill_id: guarantee_balance_aggregate
input:
  subject_scope: A集团及其子公司
  time_point: 2024-12-31
  document_facts_ref:
    - doc_12_table_3
    - doc_17_section_5
rules:
  - 只统计在指定时点有效的担保关系
  - 同一担保关系重复出现时只能计一次
  - 金额单位统一为人民币元
allowed_tools:
  - fact_retriever
  - entity_resolver
  - calculator
output_schema:
  type: object
  fields:
    - final_value
    - currency
    - evidence_list
    - calculation_trace
validation:
  - 金额单位已统一
  - 重复关系已去重
  - 每个汇总项都有证据来源
on_failure:
  - 证据不足时返回 need_more_evidence
  - 口径冲突时返回 conflict_detected
```

### 9.9 执行层如何约束 LLM

真正稳定指导 LLM，核心不是“提示词写得更花”，而是四类约束：

1. 目标约束
   - 当前步骤只允许完成单一目标，不允许顺带扩展结论
2. 上下文约束
   - 只给当前步骤相关的文档证据、规则节点、KG 子图，不给全量噪声
3. 动作约束
   - 只允许调用当前白名单里的 Skill、工具和推理模式
4. 输出约束
   - 必须按结构化格式返回，便于后续步骤消费和校验

### 9.10 Template 在执行层中的角色

在这一层里，Template 的角色不是“写回答文风”，而是：

**把 Step Contract 渲染成当前模型可执行的指令。**

所以推荐的分工是：

1. Rule
   - 定义逻辑和边界
2. KG
   - 提供结构关系和可解释上下文
3. Skill
   - 封装稳定能力单元
4. Step Contract
   - 组织当前步骤的目标、输入、约束、输出要求
5. Template
   - 把 Step Contract 编译成模型输入

### 9.11 哪些规则适合下沉成 Skill

不是所有规则都要直接变成 Skill。建议只有满足以下条件的规则才下沉：

1. 高频
2. 稳定
3. 边界清楚
4. 可参数化
5. 可重复调用

因此可以形成这样一个演进路径：

**候选规则 -> 正式规则 -> 模板化步骤 -> Skill 化能力**

### 9.12 推荐的执行层组件

建议至少拆成五个组件：

1. `Planner`
   - 把用户问题转成步骤图
2. `Context Builder`
   - 从规则库、KG、事实库中裁剪当前步骤最小上下文
3. `Prompt Compiler`
   - 把规则、Skill、上下文编译成步骤提示词
4. `Tool / Skill Router`
   - 决定调用哪个工具、哪个 Skill
5. `Validator`
   - 检查输出格式、证据完整性、数值和口径一致性

### 9.13 执行层与反馈层的连接

这一层不仅负责执行，还要把运行时信息回流给规则库，包括：

1. 哪条规则被命中
2. 哪个 Skill 被调用
3. 哪一步失败最多
4. 哪种边界条件经常导致冲突
5. 人工修正集中发生在哪些环节

这些反馈会直接决定：

1. 哪些规则需要拆分或合并
2. 哪些 Skill 可以继续沉淀
3. 哪些模板需要优化
4. 哪些问题仍然缺少稳定规则

### 9.14 一句话结论

规则库解决的是“系统学会了什么”，执行编排层解决的是“系统如何把学会的东西稳定执行出来”。

因此，执行编排层是规则资产落地为实际生产力的关键桥梁。

## 10. 数据模型建议

建议至少有以下核心表或对象：

### 10.1 `Case`

存储规则学习样本。

### 10.2 `Rule`

存储正式规则与候选规则。

### 10.3 `RuleVersion`

记录某条规则的版本、修改原因、审核人、发布时间。

### 10.4 `RuleCaseLink`

记录规则来源于哪些案例，被哪些案例验证过。

### 10.5 `ExecutionTrace`

记录一次问题处理的完整轨迹。

### 10.6 `Feedback`

记录人工修正、失败原因、召回错误、步骤缺失、边界补充等。

### 10.7 `PromptTemplate`

存储步骤级模板，支持按模型类型渲染。

### 10.8 `DocumentFact`

存储从文档抽取出来的结构化事实及原始证据定位。

## 12. 团队协作与样本生产机制

### 12.1 同学们可以做的工作

这部分非常适合组织团队协作。可以拆成三类：

1. 客户问题仿真
   - 模拟真实业务问法
2. 标准答案整理
   - 给出正确答案、证据和出处
3. 解题过程拆解
   - 把问题拆成步骤，描述正确做法

### 12.2 建议的样本任务模板

每个样本至少填写：

1. 问题原文
2. 问题类型
3. 文档类型
4. 涉及实体
5. 需要抽取的字段
6. 标准答案
7. 证据片段
8. 解题步骤
9. 计算或判断逻辑
10. 常见错误点
11. 适用范围和边界

### 12.3 审核机制

建议采用两层审核：

1. 案例审核
   - 看答案、证据、步骤是否正确
2. 规则审核
   - 看这个案例能否被抽象成稳定规则

## 13. 风险与控制点

### 风险 1：规则过碎，难维护

应对：

- 建立规则粒度标准
- 对相似规则做合并和继承
- 做规则版本管理

### 风险 2：自动学习产生大量噪声规则

应对：

- 候选规则必须经人工审核后入库
- 记录规则命中率和成功率
- 定期淘汰低质量规则

### 风险 3：模型执行不稳定，数值容易出错

应对：

- 计算尽量交给工具
- LLM 负责理解和解释
- 中间结果要做结构化校验

### 风险 4：业务问题过于多样，初期抽象困难

应对：

- 先做高频问题类型
- 先从案例归纳共性步骤
- 先跑通闭环，再逐步扩展

---

## 14. 最终结论

这个系统最合理的方向，不是做一个“更强的金融 RAG”，而是：

**构建一个以规则资产为核心的金融文档问答系统，让系统从案例中学习解题方法，把方法沉淀为规则库和知识图谱，再把这些规则编排成 prompt 与工具链，驱动大模型稳定完成复杂问答、计算和关系分析任务。**

对于当前阶段，最重要的不是先把最终问答页面做得多完整，而是先把最核心的生产线做出来：

**案例 -> 规则 -> 召回 -> 执行 -> 反馈 -> 规则更新**

只要这条线跑通，后续的图谱展示、模板体系、Agent 编排、复杂前端都可以在此基础上持续扩展。

## 15. 端到端完整链路

下面给出系统从“经验产生”到“能力复用”再到“持续增强”的完整链路。

### 15.1 一句话总链路

**客户问题/人工案例 -> 样本整理 -> 规则归纳 -> 规则审核发布 -> KG 组织 -> Template/Skill 编译 -> 新问题进入 -> 规则召回 -> 执行编排 -> LLM/工具执行 -> 结果校验 -> 答案输出 -> 人工修正/反馈 -> 规则更新 -> 系统能力增强**

### 15.2 分阶段完整链路

#### 第一阶段：经验采集层

1. 真实客户提出问题，或团队人工仿真客户问题
2. 收集对应文档、标准答案、证据出处、解题步骤
3. 把这些内容整理成标准化规则学习样本
4. 对样本进行案例审核，确保答案、证据、步骤一致

产出：

- `Case`
- 文档证据
- 解题步骤
- 边界条件

#### 第二阶段：规则学习层

1. 从多个样本中识别共性的处理逻辑
2. 把处理逻辑抽象成候选规则
3. 补充规则的适用范围、不适用边界、输入要求、校验方式
4. 人工审核后形成正式规则版本

产出：

- `Candidate Rule`
- `Published Rule`
- `RuleVersion`
- `RuleCaseLink`

#### 第三阶段：知识组织层

1. 把规则、问题类型、意图、文档类型、字段、步骤、工具组织成 KG
2. 建立规则之间的相似关系、依赖关系、继承关系
3. 把规则映射到步骤模板和执行能力单元

产出：

- `KG Subgraph`
- `PromptTemplate`
- `Skill Definition`

#### 第四阶段：能力编译层

1. 把正式规则编译成步骤级 Template
2. 把高频、稳定、可参数化的规则下沉成 Skill
3. 为每个 Skill 定义输入、输出、约束、校验、失败策略
4. 生成运行时可调用的 Step Contract 结构

产出：

- `Template`
- `Skill`
- `Step Contract`

#### 第五阶段：在线执行层

1. 新用户提出问题
2. 系统进行问题理解，识别问题类型、意图、约束条件
3. 从文档事实库中召回相关证据
4. 从规则库和 KG 中召回相关规则和 Skill
5. Planner 生成本次问题的步骤计划
6. Context Builder 为每一步构造最小上下文
7. Prompt Compiler 把 Step Contract 渲染成模型输入
8. Tool/Skill Router 决定当前步骤交给 LLM 还是工具
9. LLM/工具按步骤执行
10. Validator 对每一步结果做格式、证据、数值、口径校验
11. 形成最终答案、证据链和过程说明

产出：

- `ExecutionTrace`
- 最终答案
- 证据链
- 计算过程

#### 第六阶段：反馈回流层

1. 记录本次命中了哪些规则和 Skill
2. 记录哪些步骤成功，哪些步骤失败
3. 记录人工修正发生在哪些环节
4. 判断是更新旧规则、补边界，还是形成新规则
5. 将新的经验重新写回规则库和 Skill 库

产出：

- `Feedback`
- 规则修订
- Skill 修订
- 新案例入库

### 15.3 图示版链路

```text
[客户问题 / 人工仿真问题]
        -> [案例整理]
        -> [规则学习样本 Case]
        -> [候选规则生成]
        -> [规则审核]
        -> [正式规则库]
        -> [知识图谱 KG]
        -> [Template 编译]
        -> [Skill 编译]
        -> [新问题进入]
        -> [问题理解]
        -> [文档事实召回]
        -> [规则/KG/Skill 召回]
        -> [Planner 生成步骤计划]
        -> [Step Contract 构建]
        -> [Prompt Compiler 渲染]
        -> [LLM / Tool / Skill 执行]
        -> [Validator 校验]
        -> [答案 + 证据 + 过程输出]
        -> [人工修正 / 系统反馈]
        -> [规则更新 / Skill 更新 / 新案例沉淀]
        -> [系统能力增强]
```

### 15.4 最关键的闭环

如果只抓最核心的一条主线，可以压缩成：

**案例 -> 规则 -> KG/Template/Skill -> 执行 -> 反馈 -> 规则增强**

这条链路里：

1. `案例` 负责提供经验
2. `规则` 负责沉淀经验
3. `KG/Template/Skill` 负责组织和表达经验
4. `执行层` 负责把经验转化成生产力
5. `反馈层` 负责让系统持续变强

### 15.5 商业与技术的统一点

这条链路同时解释了系统的商业价值和技术创新：

1. 商业上，规则库积累的是行业经验和交付能力
2. 技术上，最难的是把案例稳定转成规则，并让规则稳定驱动执行
3. 一旦闭环跑通，案例越多，规则越多，Skill 越多，系统能力就会持续增强

### 15.6 一句话结论

**这不是一条“问答调用链”，而是一条“经验资产化链路”：把人的业务经验，持续转化为规则、Skill 和可执行能力，再反过来提升系统解决问题的能力。**
