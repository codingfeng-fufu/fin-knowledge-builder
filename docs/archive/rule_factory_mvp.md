# 金融文档问答系统一期文档
## 规则工厂 MVP + 规则学习样本

## 1. 文档目标

本阶段不先做完整的金融问答产品，而是先做系统最核心的能力：

1. 把案例稳定沉淀成规则库
2. 让规则可以被检索、复用、更新
3. 建立“越用越强”的持续学习闭环

一句话定义：

**本期 MVP 的目标，是验证“案例 -> 规则 -> 新问题复用 -> 反馈更新”这条闭环是否成立。**

---

## 2. 一期范围

本期只聚焦两部分：

1. **规则工厂 MVP**
   负责规则的生成、审核、发布、召回、更新。

2. **规则学习样本**
   负责定义人工案例的标准格式，让案例能被系统学习和抽象。

本期不作为重点的内容：

- 不先追求完整前端产品
- 不先追求复杂 Agent
- 不先追求全量金融场景覆盖
- 不先追求完全自动学习
- 不先追求复杂知识图谱技术实现

---

## 3. 规则工厂 MVP

### 3.1 核心定位

规则工厂不是回答问题的最终产品，而是系统的“知识生产线”。

它的职责是：

1. 接收人工整理的案例
2. 自动生成候选规则
3. 经过人工审核后发布
4. 在新问题中召回已有规则
5. 根据执行结果和人工修正持续更新规则

### 3.2 核心闭环

规则工厂 MVP 要跑通下面这条链路：

**案例入库 -> 候选规则生成 -> 人工审核 -> 规则发布 -> 新问题命中 -> 执行反馈 -> 规则更新**

### 3.3 MVP 必须具备的能力

#### 1. 案例库

存储人工整理的业务案例，包括问题、答案、证据、步骤、修正信息。

#### 2. 规则 Schema

定义规则的统一结构，至少包括：

- 规则名称
- 适用问题类型
- 触发条件
- 适用文档类型
- 输入要求
- 处理步骤
- 校验方式
- 输出格式
- 版本号
- 审核状态

#### 3. 候选规则生成

基于案例自动生成规则草稿。先不要求完全准确，但要求结构稳定、便于审核。

#### 4. 规则审核发布

人工确认规则是否可复用、边界是否清楚，再决定是否发布。

#### 5. 规则召回

新问题进入后，系统根据问题类型、关键词、约束条件、文档类型召回合适规则。

#### 6. 反馈回流

如果规则使用后效果不好，或者人工修改了答案与步骤，系统要能把这些反馈沉淀回规则库。

### 3.4 MVP 的最小数据对象

#### `Case`

表示一个学习样本。

核心字段：

- `case_id`
- `question_text`
- `question_type`
- `document_types`
- `gold_answer`
- `evidence_list`
- `solution_steps`
- `reviewer`
- `review_status`

#### `Rule`

表示一条可复用规则。

核心字段：

- `rule_id`
- `rule_name`
- `applicable_question_types`
- `trigger_signals`
- `applicable_documents`
- `required_inputs`
- `steps`
- `validation_rules`
- `output_schema`
- `version`
- `status`

#### `RuleCaseLink`

表示规则和案例的关系。

核心字段：

- `rule_id`
- `case_id`
- `relation_type`

例如：来源案例 / 验证案例。

#### `ExecutionTrace`

表示一次规则使用记录。

核心字段：

- `trace_id`
- `input_question`
- `matched_rules`
- `execution_steps`
- `final_result`
- `manual_correction`
- `failure_reason`

### 3.5 MVP 的工作流程

#### 1. 案例录入

人工提交一批标准化案例。

#### 2. 候选规则生成

系统基于案例生成规则草稿。

#### 3. 人工审核

审核规则是否：

- 可复用
- 步骤完整
- 边界清晰
- 值得发布

#### 4. 规则发布

审核通过后进入正式规则库。

#### 5. 新问题匹配

当新问题进来时，系统召回相关规则并形成处理步骤。

#### 6. 执行与反馈

执行后记录效果，若人工修正，则沉淀为规则更新依据。

#### 7. 规则迭代

更新规则版本，补充适用范围、边界条件或校验逻辑。

### 3.6 MVP 的成功标准

本期 MVP 成功，不以“回答所有问题”为标准，而以以下指标为标准：

1. 能沉淀出第一批正式规则
2. 每条规则都能追溯到来源案例
3. 新问题能召回已有规则
4. 人工修正能回流形成规则更新
5. 第二次处理类似问题时，效率和稳定性明显提高

## 4. 规则学习样本

### 4.1 定义

规则学习样本，不是简单的问答对，而是：

**可追溯、可复现、可抽象成规则的业务案例。**

它必须同时回答 5 个问题：

1. 用户问了什么
2. 正确答案是什么
3. 证据在哪里
4. 这个答案是怎么做出来的
5. 这套做法能不能抽象成规则

### 4.2 样本必须包含的内容

每个样本至少包括：

- 问题原文
- 问题类型
- 文档类型
- 标准答案
- 证据片段
- 解题步骤
- 计算或判断逻辑
- 校验点
- 适用范围
- 不适用边界

### 4.3 样本字段建议

#### 基础信息

- `case_id`
- `title`
- `source_type`
  例如：`synthetic` / `real_desensitized`
- `question_text`
- `question_paraphrases`

#### 问题标签

- `question_type`
  例如：数值计算 / 政策判断 / 关系分析
- `intent`
  例如：求和 / 筛选 / 比较 / 穿透 / 去重
- `document_types`
- `target_entities`
- `target_metrics`
- `constraints`

#### 标准答案

- `gold_answer`
- `answer_format`
- `evidence_list`

#### 解题过程

- `solution_steps`
- `formula_or_logic`
- `validation_checks`
- `common_errors`

#### 规则抽象信息

- `candidate_rule_name`
- `applicable_scope`
- `non_applicable_scope`
- `reviewer`
- `review_status`

### 4.4 最关键字段：解题步骤

样本能不能变成规则，关键看 `solution_steps` 是否完整。

建议每一步都写清：

- 这一步做什么
- 输入是什么
- 输出是什么

示例：

```yaml
solution_steps:
  - step_id: 1
    action: 识别主体范围
    input: 用户问题
    output: A集团及其子公司名单
  - step_id: 2
    action: 抽取担保记录
    input: 文档正文和表格
    output: 担保人、被担保人、余额、日期
  - step_id: 3
    action: 过滤时间条件
    input: 担保记录 + 2024年末
    output: 有效担保记录
  - step_id: 4
    action: 去重并汇总余额
    input: 有效担保记录
    output: 最终担保余额
```

### 4.5 样本类型

为了让规则库更稳，样本不能只有“标准题”，建议分成 4 类：

#### 1. 标准样本

最典型、最容易抽象规则的问题。

#### 2. 改写样本

同一规则、不同问法，用来训练召回和鲁棒性。

#### 3. 边界样本

带特殊口径、特殊约束的问题，用来补边界条件。

#### 4. 反例样本

看起来相似，但实际上不能套同一规则的问题，用来防止误召回。

### 4.6 样本生产角色

建议至少分 3 个角色：

#### 出题人

模拟真实客户问法，保证问题自然、业务化。

#### 解题人

给出标准答案、证据和步骤。

#### 审核人

确认答案、证据、步骤和规则抽象是否正确。

如果条件允许，再增加：

#### 规则抽象人

专门负责把样本转成候选规则。

### 4.7 样本验收标准

一个样本要能入库，至少满足：

1. 问法真实，不是教材式题目
2. 答案明确
3. 证据可定位到文档
4. 步骤可复现答案
5. 适用范围和边界清楚
6. 能支持规则抽象

## 5. 一期建议产出

本期建议交付以下成果：

1. 一套规则 Schema
2. 一套样本标注规范
3. 一批人工学习样本
4. 一批候选规则
5. 一批审核后的正式规则
6. 一条完整的反馈回流链路

---

## 6. 一句话总结

本期 MVP 不先证明“系统会不会答题”，而是先证明：

**系统能不能把金融业务案例持续转化成规则资产，并在后续问题中重复利用这些资产。**
