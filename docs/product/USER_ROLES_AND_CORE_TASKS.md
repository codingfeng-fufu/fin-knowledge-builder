# 用户角色与核心任务

> 目的：把“谁在用这个系统、他们到底要完成什么”先说清楚。
> 原则：按真实角色拆，不按页面拆。

## 1. 角色结论

当前系统至少存在 4 类角色，但不是每类都应该被当成 V1 主用户。

| 角色 | 是否是 V1 主用户 | 主要入口 |
|---|---|---|
| 求解用户 / 分析师 | 是 | `/workspace` |
| 方法审核人 | 是，但属于内部用户 | `/ops` |
| 平台运营 / 规则工程师 | 是，但属于内部支持角色 | `/ops` + 脚本/API |
| 演示 / 讲解 / 管理层观众 | 否 | `/workflow` / `/prototype` |

## 2. 角色一：求解用户 / 分析师

### 他们是谁

- 金融分析师
- 研究员
- 需要基于材料快速得出处理意见的人

### 他们的核心任务

当我拿到一份金融文档并有一个具体问题时，我希望系统：

- 快速给出一个可以复核的答案
- 告诉我答案依据是什么
- 告诉我当前结果是否足够可信
- 如果不能回答，明确告诉我下一步怎么办

### 他们不关心什么

默认情况下，他们不关心：

- rule graph
- meta rule
- candidate draft
- review checklist
- version rollback

这些对平台很重要，但不是他们的主任务。

### 他们的成功标准

- 能在合理时间内拿到答案
- 能看到关键证据
- 能区分“已回答”和“需要人工复核”
- 不需要理解系统内部资产结构也能完成任务

### 当前产品风险

- 页面里仍然残留很多平台叙事
- `exploration` 时内部动作暴露过多
- 默认样本和链路稳定性不一定适合第一次体验

## 3. 角色二：方法审核人

### 他们是谁

- 审核规则草稿的人
- 决定某个方法是否值得留下来的人

### 他们的核心任务

当系统在一次求解中没有稳定命中已有方法时，我需要：

- 看本次问题和材料
- 看为什么没命中稳定规则
- 看系统形成了什么 draft
- 判断这个方法是否可复用
- 决定 approve 还是 reject

### 他们的成功标准

- 审核对象信息完整
- provenance 清楚
- 可以快速判断这个方法是否值得发布
- reject 之后能继续驱动下一轮优化

### 他们真正需要看的内容

- 问题与材料摘要
- route decision
- feedback 类型
- draft 内容
- review checklist
- 历史版本与回滚信息

## 4. 角色三：平台运营 / 规则工程师

### 他们是谁

- 维护规则库健康的人
- 管版本、回滚、资产视图、图谱的人

### 他们的核心任务

当平台持续沉淀方法时，我需要：

- 监控 `workspace_run / feedback / draft / review / version`
- 发现重复规则、冲突规则、低质量规则
- 管发布、回滚和资产合并
- 维护平台整体规则质量

### 他们的成功标准

- 规则库结构可控
- 版本关系清晰
- rollback 可执行
- 资产视图能支持后续 retrieval/runtime

### 他们和审核人的区别

- 审核人关心“这条方法值不值得进库”
- 运营/规则工程师关心“整个库是否健康、可维护、可扩展”

## 5. 角色四：演示 / 讲解 / 管理层观众

### 他们是谁

- 内部汇报对象
- demo 观众
- 需要理解系统价值但不直接操作系统的人

### 他们的核心任务

他们不是来完成业务问题的，而是来理解：

- 系统在做什么
- 为什么这不是普通 Chat with PDF
- 它和方法沉淀之间是什么关系

### 对应页面

- `/workflow`
- `/prototype`

### 产品结论

这类角色存在，但不应反过来主导 V1 的核心交互设计。

## 6. 页面与角色映射

| 页面 | 主要服务对象 | 页面职责 |
|---|---|---|
| `/workspace` | 求解用户 / 分析师 | 完成一次求解，给出答案、依据、状态 |
| `/workflow` | 演示用户 / 内部讲解对象 | 解释这次求解是怎么形成的 |
| `/ops` | 方法审核人 / 平台运营 / 规则工程师 | 处理沉淀对象、审核、发布、回滚 |
| `/prototype` | 演示用户 / 内部研发 | 展示能力，不是主产品路径 |

## 7. V1 每类角色该看到什么

### 求解用户 / 分析师

应该看到：

- 问题输入
- 文档上传
- 最终答案
- 证据
- 当前状态
- 必要时的“需要人工复核”

不应该默认看到：

- draft id
- review task id
- rule graph
- version 信息
- publish / rollback 动作

### 方法审核人

应该看到：

- feedback
- draft
- review checklist
- provenance
- approve / reject

### 平台运营 / 规则工程师

应该看到：

- 全量资产视图
- 规则图谱
- 版本
- rollback
- 历史关系

## 8. 三条典型任务路径

### 路径 A：求解用户正常完成任务

```text
/workspace
-> 上传文档
-> 输入问题
-> direct_match
-> 得到答案和证据
-> 结束
```

### 路径 B：求解用户进入不确定状态

```text
/workspace
-> 上传文档
-> 输入问题
-> exploration / needs_more_context
-> 告知“需要补充材料”或“需要人工复核”
-> 转交内部流程
```

### 路径 C：内部方法审核

```text
/ops
-> 查看 feedback / draft / review
-> approve or reject
-> publish or rerun
```

## 9. 角色结论

V1 最关键的产品动作只有两个：

- 帮求解用户更稳定地完成一次问题求解
- 帮内部审核人更有效地处理 exploration 产物

除此之外的页面和能力，都不应该抢走这两个角色的主任务。
