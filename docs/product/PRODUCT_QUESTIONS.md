# 产品问题清单

> 目的：不是继续加功能，而是先回答“这到底是不是一个成立的产品”。
> 依据：当前代码、实际走查结果、`PRODUCT_CHAIN_WALKTHROUGH.md`。

## 1. 当前判断

当前系统里其实混着三种不同性质的产品：

- `/workspace`：问题求解工作台
- `/workflow`：过程解释与对外演示页
- `/ops`：方法沉淀与治理后台

技术上它们已经连起来了。

但产品上，三者还没有被明确拆成：

- 谁是主入口
- 谁是主用户
- 谁承担主要价值
- 哪些能力该给用户看
- 哪些能力只应该留在内部

所以当前最大问题不是“缺功能”，而是“产品定义不够收敛”。

## 2. P0 问题

这些问题不先定，后续页面、默认样本、指标和路线图都会继续漂。

| 问题 | 当前现状 | 建议答案 |
|---|---|---|
| 我们卖的到底是什么 | 口径在“求解工作台”和“方法资产平台”之间摇摆 | 对外先定义为“带证据的金融文档求解工作台”，把“方法资产平台”定义为内部能力层 |
| 谁是主用户 | 现在把提问者、审核者、规则运营者混在一起讲 | V1 主用户应是“求解用户/分析师”，审核人和平台运营是内部角色 |
| 用户在 `exploration` 时应得到什么 | 现在同时有探索入口、审核动作、规则草稿、平台叙事 | 对普通用户只承诺“需要人工复核”或“进入内部处理”，不要把平台内脏完整暴露出来 |
| `direct_match` 和 `exploration` 是不是同一种产品体验 | 当前页面都挂在同一个主流程里 | 不是。`direct_match` 是用户任务完成路径；`exploration` 是平台学习路径 |
| 首次体验要不要继续用当前默认样本 | 当前默认样本偏 `equity_research`，链路更重，体验不稳定 | V1 默认体验应该回到最稳、最短路径的 `fund_nav_warning` |
| 产品成功指标是什么 | 现在没有稳定的产品级指标口径 | 先定 3 个：可用答案率、需要人工复核率、首次可辩护答案耗时 |

## 3. P1 问题

这些问题不是现在不重要，而是应该放在产品定义稳定之后。

| 问题 | 当前状态 | 建议处理 |
|---|---|---|
| 是否要对外讲“多智能体生规则” | 当前叙事很强，但对外价值不够直接 | 暂时只作为内部能力，不作为 V1 外部卖点 |
| 是否要把 `workflow` 做成正式产品页 | 当前更像解释页和 demo 支撑页 | 先保留为演示/内部说明页，不放进核心用户主流程 |
| 是否要对外暴露 Rule Graph | 当前 `/ops` 已经展示 | 暂不对普通用户暴露，先留给内部治理 |
| 是否要承诺“所有金融场景都能处理” | 当前多场景都在仓库里，但成熟度明显不同 | V1 只承诺少数稳定场景，其他场景明确为 demo/实验能力 |
| 是否要把方法资产平台单独产品化 | 当前具备雏形，但角色和场景更偏内部 | 放到 V2 之后，再决定是否独立成平台型产品 |

## 4. 现在最真实的产品矛盾

### 4.1 用户想要答案，系统想要沉淀方法

这是当前最根本的矛盾。

用户进入 `/workspace` 时的自然目标是：

- 给我答案
- 告诉我依据
- 告诉我能不能信

但系统内部最重的能力投入是在：

- 规则沉淀
- draft/review/publish
- 资产治理

这两者都对，但不是一个层级的问题。

如果不拆开，用户会感觉产品过重，平台会感觉入口不稳定。

### 4.2 现在的“成功”定义不清楚

对于 `direct_match`：

- 成功是“给出可用答案”

对于 `exploration`：

- 对平台来说成功可能是“生成了 draft/review”
- 但对用户来说，这其实只是“还没解决”

这意味着我们需要两个成功定义：

- 用户成功
- 平台学习成功

不能继续用一个结果状态同时承担这两件事。

### 4.3 首次体验被平台目标绑架了

当前文档、页面、默认样本、解释页都在强调“把答案变成方法”。

这在内部叙事上成立。

但对第一次进产品的人来说，真正需要先被证明的是：

- 这个系统能不能稳定帮我解决一个具体问题

如果第一印象就是 exploration、draft、review、multi-agent，那会让人觉得它更像研究系统，不像产品。

## 5. 这一周必须定下来的 5 个产品决策

1. 对外产品定义先收敛为“求解工作台”，还是继续强推“方法资产平台”。
2. 主用户先定成“分析师/业务用户”，还是“规则运营/审核人员”。
3. `exploration` 是否继续在主页面暴露完整的 draft/review 动作。
4. 首页默认体验是否回切到稳定 case。
5. 产品指标是否正式分成“用户任务指标”和“平台沉淀指标”两组。

## 6. 当前推荐答案

如果今天必须做一次产品收敛，我的建议是：

- 对外产品名义上先做“带证据的金融文档求解工作台”
- 对内保留“方法资产平台”作为系统能力和长期方向
- `/workspace` 只对用户承诺求解和证据
- `/workflow` 只承担解释和演示
- `/ops` 只给内部角色使用
- `exploration` 在用户视角上表达为“需要人工复核”，而不是“平台正在自动学会新方法”

这不是降级目标，而是把产品承诺和系统目标拆开。
