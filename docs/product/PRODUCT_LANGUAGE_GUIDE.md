# 产品语言指南

> 目的：统一当前产品界面的用户语言，避免同一件事在不同页面里被说成不同的词。

## 0. 官方 Hero 口径

主标题：

**让每一次回答，都成为下一次的能力**

副标题：

**把答案变成规则，把规则变成系统持续进化的起点。**

使用建议：

- `/workspace`：直接使用
- `/demo`：可直接使用，或在不改变核心含义的前提下做轻微演绎
- `/workflow` 与 `/ops`：不直接复用主标题，但应保持“答案 -> 规则 -> 持续进化”这条叙事一致

避免把这句改写成过于技术化的版本，例如：

- “多智能体规则生成平台”
- “基于规则库的执行编排系统”

这些可以是内部描述，不适合替代 Hero 口径。

## 1. 总原则

面向普通用户时，优先使用：

- 答案
- 依据
- 当前状态
- 补充材料
- 人工复核
- 内部处理

尽量少用或只在内部页面使用：

- 路径
- 路由
- exploration
- draft
- publish
- rollback
- rule graph

## 2. 推荐词汇

| 场景 | 推荐说法 | 少用说法 |
|---|---|---|
| 用户已得到稳定结果 | 已生成结论 | direct_match |
| 多规则拼接完成 | 组合求解 | rule_composable |
| 还缺信息 | 等待补充材料 | needs_more_context |
| 当前答不稳，转交内部 | 进入内部处理 | exploration |
| 答案来自哪一层 | 结果来源 | 答案引擎 |
| 为什么出现这个结果 | 当前状态 / 处理方式 | 路由 / 路径 |
| 内部在看什么 | 内部治理 / 内部记录 | 方法沉淀 / 资产闭环 |

## 3. 页面级口径

### `/workspace`

对外主入口。

优先使用：

- 最终建议
- 结果概览
- 关键证据
- 当前状态
- 判断依据
- 状态说明
- 内部记录

### `/workflow`

说明页。

优先使用：

- 系统说明
- 这次答案是怎么形成的
- 已有方法 / 进入内部处理

### `/ops`

内部页。

优先使用：

- 内部治理
- 审核
- 发布
- 回滚

## 4. 需要持续避免的混用

- 同一页面里同时说“路径”和“当前状态”
- 对普通用户同时暴露“人工复核”和“同意接入方法库”
- 在主入口上把“内部治理”语言和“用户求解”语言混成一套
