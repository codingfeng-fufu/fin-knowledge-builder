# V1 PRD

> Hero 主标题：**让每一次回答，都成为下一次的能力**
>
> Hero 副标题：**把答案变成规则，把规则变成系统持续进化的起点。**

## 1. 产品名称

金融文档求解工作台 V1

副口径：

由规则资产平台支撑的、带证据的金融文档求解工作台。

## 2. 背景

当前系统已经具备两层能力：

- 对用户侧：基于金融文档完成问题求解
- 对平台侧：把一次求解继续沉淀成方法资产

问题在于，这两层能力当前混在同一个产品叙事里，导致：

- 首次体验更像研究系统，不像产品
- 用户价值和平台价值混在一起
- 页面文案、默认样本、内部动作暴露过多

V1 的目标不是把所有能力都包装成一个平台产品，而是先把用户可感知价值收敛成一个成立的主入口。

## 3. V1 产品定义

V1 对外定义为：

**带证据的金融文档求解工作台**

它向用户承诺三件事：

1. 用户上传文档并提问后，系统尽量给出明确答案。
2. 系统同时展示关键证据和当前状态。
3. 如果当前无法稳定回答，系统会明确标记为“需要补充材料”或“需要人工复核”。

V1 不对普通用户承诺：

- 规则治理
- draft/review/publish
- 多智能体生规则
- 规则图谱

这些属于内部能力层。

## 4. 目标用户

主用户：

- 金融分析师
- 研究员
- 需要基于文档快速形成处理意见的人

内部用户：

- 方法审核人
- 规则运营 / 平台工程师

演示用户：

- 内部汇报对象
- 需要理解系统价值的人

## 5. 用户核心任务

### 主用户任务

当我拿到一份金融文档和一个具体问题时，我希望：

- 快速得到答案
- 看到支撑答案的证据
- 判断当前结果是否足够可信
- 如果答不稳，知道下一步该怎么办

### 内部用户任务

当系统答不稳时，我希望：

- 自动接住 exploration 产物
- 形成 review 对象
- 决定是否发布为方法资产

## 6. V1 范围

纳入范围：

- `/workspace` 作为唯一主入口
- 推荐样本与真实上传文件两种输入方式
- direct-match 稳定求解
- evidence 展示
- 明确状态表达
- exploration 进入内部处理
- `/ops` 作为内部审核与治理页面

不纳入范围：

- 把 `/workflow` 当成主产品入口
- 对外讲多智能体规则生成
- 对普通用户暴露 rule graph / version / rollback
- 承诺所有金融场景都稳定可用

## 7. 核心产品流程

### 流程 A：正常求解

```text
进入 /workspace
-> 选择推荐样本或上传文件
-> 输入问题
-> 运行
-> 获得答案
-> 查看证据
-> 结束
```

### 流程 B：无法稳定回答

```text
进入 /workspace
-> 选择推荐样本或上传文件
-> 输入问题
-> 运行
-> 系统判断需要补充材料或人工复核
-> 用户看到明确状态
-> 问题转入内部处理
```

### 流程 C：内部治理

```text
进入 /ops
-> 查看 workspace_run / feedback / draft / review
-> approve / reject
-> publish / rollback
```

## 8. 页面定位

### `/workspace`

产品主入口。

用户只需要理解：

- 我要问什么
- 系统给了什么答案
- 依据是什么
- 当前状态是什么

### `/workflow`

解释页。

用于：

- demo
- 内部说明
- 解释系统如何形成这次解法

### `/ops`

内部治理页。

用于：

- review
- publish
- rollback
- 规则资产管理

## 9. 用户可见状态

V1 面向用户只输出三类状态：

- 已生成结论
- 需要补充材料
- 需要人工复核

技术状态如：

- direct_match
- rule_composable
- exploration
- binding_status

保留在详情层或内部层。

## 10. 默认体验

V1 默认样本应使用稳定路径：

- `workspace/fund_docx_direct_warn`

理由：

- 场景更稳定
- 可解释性强
- 更适合第一次体验

同时保留：

- 用户手动触发，不自动执行
- 推荐样本可直接运行
- 上传本地文件后优先使用本地文件

## 11. 成功指标

用户侧指标：

- 可用答案率
- 首次可辩护答案耗时
- 需要人工复核率

平台侧指标：

- exploration 自动形成 review 的比例
- 审核通过率
- 发布规则版本数

## 12. 验收标准

V1 至少满足：

1. 稳定场景下，`/workspace` 能给出答案和关键证据。
2. 推荐样本可直接运行，不要求用户额外上传演示文件。
3. 页面能清楚区分“已回答”和“需要人工复核”。
4. exploration 产物能自动进入内部治理链。
5. 页面文案和真实交互行为一致。

## 13. 发布前动作

1. 默认样本切回 fund warning。
2. 主入口文案从“平台叙事”收回到“答案与依据”。
3. 将内部治理动作从 `/workspace` 主视图移除或弱化。
4. 补齐文档与代码行为不一致处。
5. 用 golden cases 做一次 direct-match 与 exploration 的回归。

对应执行清单见：

- `V1_LAUNCH_CHECKLIST.md`

## 14. 一句话总结

V1 不是完整的平台产品发布。

V1 是：

**一个由规则资产平台支撑的、对用户承诺答案、依据和明确状态的金融文档求解工作台。**
