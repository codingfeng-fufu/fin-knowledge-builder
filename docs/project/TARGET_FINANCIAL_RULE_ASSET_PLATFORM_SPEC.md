## 1. 平台核心定位

本平台是一款以**通用金融规则库**为核心资产的智能处理系统。

平台不以“单一案例求解”为核心，而以“**某一类金融问题的标准化处理范式**”为核心。

规则库中的每条规则都应满足以下要求：

- 以文字形式存储
- 面向一类金融问题，而不是单一具体样例
- 描述标准化处理逻辑、步骤、判断条件、证据要求和输出形式
- 可被后续 Skill Creator 转化为运行期 Skill
- 可被大模型、工具、代码执行器和人工审核共同消费

平台的价值不在于替代大模型，而在于补足普通大模型在以下方面的不足：

- 数值计算
- 复杂业务逻辑执行
- 规则条件判断
- 结构化证据引用
- 图表和可视化生成
- 面向金融业务的稳定处理流程

一句话：

**平台的核心资产是通用规则库，核心能力是把规则库、PDF 文档、问题输入和大模型执行能力整合成一套稳定的金融问题处理系统。**


## 2. 目标系统组成

目标系统应由以下核心部分构成：

### 2.1 核心规则库

作用：

- 存储通用金融处理规则
- 覆盖多类金融问题
- 作为平台最核心、最可积累、最可复用的资产

规则库中的规则应具有：

- 规则名称
- 规则适用范围
- 规则输入要求
- 规则处理步骤
- 规则输出要求
- 证据与校验要求
- 版本信息
- 人工审核状态

### 2.2 PDF 阅读器 Agent

作用：

- 对用户上传的金融 PDF 进行全文理解
- 根据用户问题聚焦相关内容
- 输出可用于后续规则匹配与执行的上下文

该模块应基于 **Anthropic Claude Code 风格的 PDF 阅读能力**构建，但平台实现不应受限于单一模型厂商。

它的目标不是“抽文本”，而是“读懂文档并形成可推理的上下文”。

### 2.3 上下文构建模块

作用：

- 将用户问题（Query）
- PDF 阅读结果
- 大模型初步理解

融合成结构化上下文（Context）

该 Context 必须足够支撑：

- 规则检索
- 规则适配判断
- Skill 生成
- 超级 Agent 执行

### 2.4 规则匹配与验证模块

作用：

- 基于 Context 从规则库中检索候选规则
- 对候选规则进行适配性判断
- 确认当前问题是否存在可用规则

该模块不是只做召回，而是要做：

- 规则候选检索
- 规则适配性验证
- 执行前判定

### 2.5 Skill Creator 模块

作用：

- 将选中的金融规则、用户问题、上下文输入到 Skill Creator
- 动态生成运行期 Skill

要求：

- Skill Creator 本身就是一个 Skill
- 生成结果以 Markdown 步骤说明为主
- 可附带脚本、工具、小程序
- Skill 是**运行期中间产物**

### 2.6 Coding 超级 Agent

作用：

- 接收生成后的 Skill
- 根据任务需要决定如何执行
- 必要时自主写代码、调工具、回溯原文、生成图表
- 输出最终用户答案

要求：

- 基于 Anthropic Claude Code 开源代码搭建
- 大模型驱动应可切换，例如：
  - 千问
  - Kimi
  - 其他兼容模型
- 执行层必须具备“做事”能力，而不仅是“生成文本”能力

### 2.7 多智能体规则生成模块

作用：

- 当规则库无匹配规则时
- 参考已有规则与上下文
- 自动生成新的通用金融处理规则

要求：

- 采用蚁群 / 蜂群式多智能体协同模式
- 不依赖单一智能体单次生成
- 允许在结果不理想时继续迭代生成

### 2.8 人工审核与规则沉淀模块

作用：

- 对新规则和测试结果进行人工审核
- 通过后入库
- 不通过时触发下一轮多智能体优化

要求：

- 人工审核是规则入库前的必经环节
- 规则库扩充必须经过审核
- 审核反馈要能反向驱动规则迭代


## 3. 目标常规流程

常规流程适用于：

**规则库中已经存在可用规则。**

### 3.1 用户输入发起请求

用户行为：

- 上传金融 PDF 文档
- 输入具体业务问题

系统产物：

- `Document`
- `Query`

这一阶段的输出只是原始请求，不做结论生成。

### 3.2 PDF 文档智能阅读与上下文提取

系统调用 PDF 阅读器 Agent，对整份 PDF 进行阅读与理解。

该阶段必须完成两件事：

1. 全文理解

- 识别整份文档的结构、主题、关键部分、图表、数字、章节与业务对象

2. 问题聚焦

- 结合 Query 找到与当前问题最相关的位置、段落、表格和证据

随后再结合大模型的初步理解，输出结构化 Context。

该 Context 至少应包含：

- 问题相关的文档位置
- 问题相关的关键内容
- 初步业务理解
- 可供规则匹配的结构化线索

这一阶段的目标不是直接回答问题，而是：

**为规则匹配与执行准备正确上下文。**

### 3.3 规则匹配与验证

系统基于 Context 在规则库中进行检索。

该阶段包括：

- 规则召回
- 规则过滤
- 规则适配性验证
- 规则可执行性确认

只有当系统判断某条规则确实适合解决当前问题时，才进入下一阶段。

这一阶段的输出应为：

- `MatchedRule`
- `RuleValidationResult`

### 3.4 Skill 生成

系统调用开源 Skill Creator 工具，将如下输入一并提交：

- `MatchedRule`
- `Query`
- `Context`

Skill Creator 输出：

- 运行期 Skill（Markdown 步骤指导）
- 可选脚本
- 可选小工具

要求：

- Skill 仅服务当前任务
- Skill 可以包含：
  - 数值计算指令
  - 文档回溯指令
  - 工具调用方式
  - 图表生成方式
  - 结果整理方式

### 3.5 超级 Agent 执行

系统将生成的 Skill 交给 Coding 超级 Agent。

超级 Agent 必须具备以下执行能力：

- 根据 Skill 自主规划执行顺序
- 写代码完成数值计算
- 必要时再次回溯 PDF 原文
- 调用脚本或工具
- 生成图表
- 组织结构化结果

这一阶段不要求 Skill 永久存在。

正确要求是：

- Skill 是运行期产物
- 用完即弃
- 不做持久化沉淀

### 3.6 输出最终结果

超级 Agent 完成执行后输出：

- 面向用户的最终回答
- 可选证据引用
- 可选图表或可视化内容

最终答案必须满足：

- 精准回答 Query
- 逻辑与规则一致
- 必要时包含原文依据
- 能处理普通大模型难以稳定完成的任务


## 4. 目标特殊流程

特殊流程适用于：

**规则库中没有匹配规则。**

### 4.1 多智能体协同生成新规则

当规则库无匹配规则时，系统启动蚁群 / 蜂群模式多智能体。

输入：

- `Query`
- `Context`
- `RuleLibrary`

目标：

- 参考已有规则逻辑
- 智能涌现出一条新的通用金融处理规则

输出：

- `NewRuleCandidate`

要求：

- 新规则不是只为当前样例服务
- 新规则必须可推广到一类问题

### 4.2 新规则测试执行

新规则候选产生后，不直接入库。

系统仍按常规流程执行：

- `NewRuleCandidate`
- `Query`
- `Context`
-> `Skill Creator`
-> 生成运行期 Skill
-> 交给超级 Agent 执行
-> 产出初步结果

这一阶段的目标是验证：

- 规则是否真的可执行
- 规则是否真的能解决当前问题

### 4.3 人工审核

人工审核需要同时判断：

- 当前结果是否合格
- 这条新规则是否有效
- 这条新规则是否具备通用性

审核结果分为两类：

#### 审核通过

表示：

- 当前结果合格
- 新规则有效
- 新规则可推广

则系统执行：

- 将该规则写入核心规则库
- 完成规则库扩充

#### 审核不通过

表示：

- 结果不合格
- 或规则不稳定
- 或规则不具备通用性

则系统执行：

- 将失败结果
- 原始 Query
- Context
- RuleLibrary
- 审核反馈

重新喂给多智能体协同模块，要求其调整优化规则并重新生成

之后重复：

- 规则生成
- Skill 转化
- 超级 Agent 执行
- 人工审核

直到得到合格规则后再入库。


## 5. 平台数据对象

目标系统中至少应存在以下核心对象：

- `Query`
- `Document`
- `Context`
- `Rule`
- `RuleLibrary`
- `MatchedRule`
- `RuleValidationResult`
- `Skill`
- `FinalAnswer`
- `NewRuleCandidate`
- `ReviewRecord`

### 5.1 `Query`

表示用户问题。

最少包含：

- 原始问题文本
- 问题类型
- 业务目标

### 5.2 `Context`

表示问题相关的文档上下文。

最少包含：

- 相关文档片段
- 关键位置
- 初步理解
- 结构化线索

### 5.3 `Rule`

表示通用金融处理规则。

必须包含：

- 适用问题类型
- 输入要求
- 处理步骤
- 输出要求
- 校验逻辑

### 5.4 `Skill`

表示运行期由规则转化得到的中间执行产物。

特点：

- 临时生成
- 用完即弃
- 不做长期持久化

### 5.5 `NewRuleCandidate`

表示多智能体新生成的规则候选。

特点：

- 未经过人工审核
- 不属于正式规则库资产
- 只能进入测试执行链


## 6. 关键设计约束

### 6.1 规则优先，而非案例优先

平台的中心应始终是规则库，不是案例库。

允许案例用于：

- 测试
-验证
-审核

但规则沉淀必须面向一类问题。

### 6.2 Skill 只是运行期中间产物

Skill 的作用是帮助当前任务执行，不应成为平台长期主资产。

平台长期主资产应是：

- RuleLibrary

不是：

- 临时 Skill 文件

### 6.3 执行层必须具备真实执行能力

如果执行层只能“生成一段文本”，则不满足目标要求。

目标执行层必须具备：

- 代码生成与运行
- 工具调用
- 证据回溯
- 图表生成
- 多步骤执行

### 6.4 新规则入库必须有人工审核

系统可以自动生成候选规则，但不能自动把未经审核的规则写入正式规则库。

### 6.5 规则生成必须可迭代

一轮生成失败后，不应中止，而应：

- 将失败结果与审核意见重新反馈给多智能体
- 引导其迭代生成更好的规则


## 7. 目标验收标准

当系统满足以下条件时，才可以视为达到目标平台形态：

### 7.1 常规流程验收

- 用户上传 PDF 并提问
- 系统能构建高质量 Context
- 能从规则库中找到可用规则
- 能将规则转化为 Skill
- 超级 Agent 能执行 Skill 并输出结果
- 最终结果可处理数值、逻辑、图表等复杂任务

### 7.2 特殊流程验收

- 无匹配规则时能触发多智能体生成新规则
- 新规则可进入测试执行链
- 人工审核后可入库
- 审核不通过时可进入迭代优化循环

### 7.3 资产层验收

- 规则库持续扩充
- 规则具备通用性
- 规则能长期复用
- Skill 不作为长期资产沉淀


## 8. 当前实现对齐原则

后续所有具体实现都应围绕以下原则推进：

1. 先确保 PDF Reader Agent 能产出高质量 Context
2. 再确保规则匹配与适配判断可靠
3. 再确保 Skill Creator 输出可执行 Skill
4. 再确保超级 Agent 具备真实执行能力
5. 最后闭合“无规则 -> 多智能体生规则 -> 人审入库”的完整循环

任何阶段的实现都不应把：

- 临时中间产物
- 当前实验性 workaround
- 局部单案例逻辑

误当成平台核心形态。
