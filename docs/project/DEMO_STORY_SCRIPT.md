# 把答案变成方法

副标题：一个把金融问题解法沉淀下来的智能工作台

## 核心口径

- 我们不是在做一个 `Chat with PDF` 工具。
- 我们想做的是：把一次次金融问题求解，沉淀成可复用的方法。
- 我们真正积累的，不是一条答案，而是解决这类问题的方式。

## 开场

大家好，我们做的不是一个普通的文档问答系统。

很多人做的是 `Chat with PDF`：把文档丢进去，问一个问题，模型给一个答案，这次就结束了。

但在金融场景里，真正有价值的，从来不只是“回答了一次问题”，而是这次问题到底是怎么被解决的。

因为金融问题往往不是单纯的信息检索，它更像是一个求解过程：
先看哪些材料，抓哪些证据，提哪些关键信息，做哪些判断，必要时还要完成计算，最后才能形成一个可复核的答案。

所以我们想做的，其实不是一个只会回答问题的系统，而是一个能把解决问题的过程留下来、复用起来、不断积累起来的智能工作台。

如果用一句话概括，就是：

**我们希望把答案变成方法。**

## 故事主线

今天我用同一份真实研报，给大家展示三个层次。

第一层，是系统能不能真正读懂文档。

第二层，是系统能不能不只抽取信息，而是真正参与解决问题。

第三层，是系统能不能把这次求解过程沉淀下来，变成下一次还能复用的方法。

这三层连起来，才是我们这个项目真正想做的事情。

## 第一层：先读懂文档

我先打开 `/workspace`。

这里的核心不是把问题直接扔给大模型，而是先让系统理解问题和材料，再去组织答案。

现在我用的是一份真实的工商银行研报。第一个问题很直接：

**这份研报对工商银行的投资评级是什么？**

这个问题看起来简单，但它想证明一件基础能力：系统到底是不是在“基于证据”回答。

在这里，系统会先读取文档，再把当前问题和文档内容建立联系，然后给出最终答案。
它识别出来的是“增持（维持）”，同时还能给出对应证据。

所以第一层我们想证明的不是“模型会不会说话”，而是：

**它能不能真正从文档里找到支撑结论的依据。**

## 第二层：不是只会抽取，而是会解题

接下来我换一个问题，但材料还是同一份研报。

这次的问题是：

**这份研报对工商银行的投资评级是“增持（维持）”。根据研报中的评级定义，“增持”表示未来 6 个月股价相对上证指数强于 5% 到 15%。请结合第 1 页收盘价 7.63 元，写一段 Python 代码计算对应的股价区间，并给出最终区间答案。**

这一步非常重要。

因为现在系统面对的已经不是一个纯抽取问题了。
它不能只把原文摘出来，而是要把文档里的定义转成一个可以执行、可以复核的求解过程。

在这个例子里，它需要完成几件事：

第一，识别当前评级是“增持（维持）”。

第二，理解研报里对“增持”的定义，不是一个模糊判断，而是一个 5% 到 15% 的区间。

第三，找到第 1 页的收盘价 7.63 元。

第四，把这些信息组织成一段清楚的 Python 代码，并算出最终结果。

最后系统给出的区间是：

**8.01 元到 8.77 元。**

这一步的意义在于：

我们证明的已经不只是“系统能读文档”，而是：

**系统能把文档里的业务定义，转成一个可复核、可计算、可解释的问题求解过程。**

这和普通问答系统的差别非常大。

普通问答系统更像是“给我一个答案”。

而我们更关心的是：

**这个答案是怎么被解出来的。**

## 第三层：把这次求解，沉淀成方法

但如果系统只做到前两层，它依然只是一个更强一点的问答工具。

真正让这个项目有价值的，是第三层。

第三层是：**把这次问题求解的方式沉淀下来。**

也就是说，系统不会把这次过程当成一次性输出。
它会保留这次真正有价值的结构：

- 这是什么类型的问题
- 适合看什么材料
- 应该关注哪些证据
- 要提取哪些关键字段
- 需要做什么计算
- 最终答案应该怎么组织

所以我们沉淀的，不是一条死规则，也不只是一个单次答案。

我们沉淀的，是：

**面对某一类金融问题时，系统应该如何解决它。**

换句话说，今天这次系统不是简单地回答了“工行这篇研报怎么理解”。

更重要的是，它开始沉淀一种可复用的方法：

以后再遇到同类的研报、同类的评级定义、同类的区间计算问题，系统就不需要从零开始，而是可以沿着这次已经形成的方法继续往前走。

## 为什么这件事重要

这也是我们和普通文档问答最大的区别。

普通系统的终点是“答完”。

我们的终点不是答完，而是：

**答完之后，系统有没有多学会一种解决问题的方式。**

如果每次回答之后，系统都能多沉淀一点方法，
那这个系统积累下来的，就不只是历史记录，
而是一套越来越完整、越来越可复用的问题求解能力。

这对于金融场景尤其重要。

因为在这个场景里，用户真正需要的不是“好像有道理的回答”，
而是一个有证据、有过程、能复核、能迁移的方法。

## 收束

所以，我们这个项目的核心，不是“让模型读 PDF”。

我们真正想做的是：

从文档理解出发，
让系统完成判断、完成计算，
并把这次求解过程沉淀成下一次还能复用的方法。

如果用一句话作为今天的总结，就是：

**我们不是在存答案，我们是在沉淀解法。**

或者更完整一点：

**我们不是在训练一个会读金融文档的助手，而是在构建一个能不断沉淀金融问题解法的系统。**

谢谢大家。

## 结尾金句备选

- 我们真正积累的，不是一条答案，而是解决这类问题的方法。
- 真正有价值的不是答对一次，而是留下怎么解。
- 普通系统回答问题，我们的系统沉淀解法。
- 我们不是在存历史结果，而是在积累未来还能复用的能力。
